    # 攻击篇

    # XSS 攻击

    XSS 攻击是当用户访问网页时，攻击者通过恶意脚本的方式执行其他操作，盗取用户的数据等。

    特点：

    1. 通常难以从UI上感知（暗地执行脚本）
    2. 窃取用户信息（cookie/token）
    3. 绘制 UI (例如弹窗)，诱骗用户点击/填写表单。
    类型 特点
    存储型 XSS 攻击者将恶意脚本上传至数据库，当用户访问该网页时，就会执行数据库中的脚本。例如，用户评论时，攻击者可以评论特定的脚本代码，如果服务器端未作处理，则会被注入恶意脚本。
    反射型 XSS 攻击者通过URL参数的方式，向服务器端发送恶意脚本。如果有用户点击了该URL链接，就会执行恶意脚本。
    DOM型 XSS 攻击者通过URL参数的方式，向JS脚本中注入恶意内容。与反射型XSS不同的是，反射型XSS需要服务器做反射，而DOM型XSS是利用客户端的漏洞进行攻击。

    **注意：**由于HTML5中并不会执行innerHTML中的script标签，所以在存储型XSS攻击中，需要利用DOM自身的特点。

    /* 存储型XSS */
/* 例如 服务端部分代码 */
const post = getPost();
document.getElementById('exp').innerHTML = post;

/* 我们在进行XSS攻击时这样上传脚本是无效的 */
<script>alert()</script>
    
/* 应该这样上传 */
<img src="x" onerror="alert(xxx)" />

    # CSRF 攻击

    攻击者首先构造一个恶意第三方网站，诱导用户点击。当用户点击后，向攻击者指定网站发起请求。如果此时用户该网站的cookie还未失效，那么攻击者就得到了用户的登录凭证。CSRF攻击可以在用户不知情的情况下，利用用户权限(cookie)，构造指定 HTTP 请求，窃取或修改用户敏感信息。

    例如，小明点击了自己一封邮件里的网址，此时小明处于登录状态，攻击者利用小明的cookie权限向小明邮箱中添加了一个过滤器，每一封小明的邮件都会优先转发给攻击者指定的人。开始小明并未发现有什么异常，一段时间后，攻击者利用邮件转移了小明的域名并索要赎回费用后，小明才发现是那封邮件里的网址有问题。这就是一次完整的CSRF攻击。

    # 注入

    SQL注入是一种常见的攻击方式，攻击者上传特定的SQL语句到服务器，服务器在执行SQL语句时，被攻击者的SQL语句干扰，执行了预期之外的SQL语句。

    也有其余形式的注入攻击：

    SSRF(服务端请求伪造)

    攻击者利用漏洞来让服务器端发送请求，从而获取到仅服务器能够访问的内网资源。

    # Dos

    类型 特点
    ReDos 利用复杂的正则表达式，消耗计算机资源，让计算机长时间的超负荷运转
    DDos 在短时间内向服务器发出大量请求，使得浏览器承受不住而崩溃
    特点：直接访问 IP，可攻击任意 API，消耗大量带宽(耗尽)

    # CSRF 的防御

    设置 Origin + Referrer：Origin 指示了请求来自于哪个站点。Referer 包含了当前请求页面的来源页面的地址。

    设置 X-Frame-Options（防御iframe标签伪造的同站）：

    The X-Frame-Options HTTP 响应头是用来给浏览器 指示允许一个页面可否在 <frame>, <iframe>, <embed> 或者 <object> 中展现的标记。站点可以通过确保网站没有被嵌入到别人的站点里面，从而避免 clickjacking 攻击。

    SameSite Cookie：用来防止CSRF攻击和用户追踪。一共有三个值，如下所示

    值 含义
    Strict Strict 代表完全禁止第三方Cookie，这样确实安全，但同时也限制了依赖Cookie的第三方应用
    Lax 仅在链接，预加载请求，GET 表单三种情况时发送Cookie
    None 禁用SameSite，任意时候均发送Cookie，不过必须要同时设置Secure属性（Cookie 只能通过 HTTPS 协议发送），不然无效
    /*设置为 None*/
Set-Cookie: widget_session=abc123; SameSite=None; Secure

    # 防御 SQL 注入

    1. 最小权限原则。
    2. 建立允许名单 + 过滤。
    3. 对 URL 类型参数进行协议、域名、ip 等限制。

    防御 reDOS

    1. 复查自己的正则表达式是否合理
    2. 利用代码扫描工具
    3. 拒绝使用用户提供的正则

    1. 流量治理

      • 负载均衡

      • API 网关

      • CDN

    2. 快速自动扩容

    3. 非核心服务降级

    # CDN劫持与SRI

    在 Web 开发中，我们经常使用 CDN 资源来加快网络请求资源的速度，但是由于CDN来自第三方服务器，所以并不安全。

    SRI 是利用hash值的方式，来检查文件是否正常。当加载文件之前，会对比加载文件的hash值和预期的hash值，只有二者相同才加载文件，否则返回请求文件错误。
